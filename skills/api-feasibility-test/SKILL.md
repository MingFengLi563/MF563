# api-feasibility-test - 接口可行性测试助手

## 技能定位

帮助非技术背景的产品经理测试 API 接口可用性，获取实际返回参数，截图官方前端页面样式，输出结构化可行性报告。

## 触发条件

当用户提到以下情况时激活此技能：
- "测试这个接口能不能用"
- "接口返回什么数据"
- "帮我看看这个 API 可不可行"
- "产品方案需要评估接口可行性"
- "这个接口的前端页面长什么样"
- "接口参数实际是什么样的"

## 核心能力

1. **接口可用性测试** — 发送测试请求，验证接口是否可用
2. **返回参数解析** — 提取实际返回字段、类型、示例值
3. **前端页面截图** — 使用浏览器自动化截取官方文档/示例页面
4. **可行性评估** — 输出 ✅/⚠️/❌ 结论 + 风险点 + 替代方案

## 执行流程

### Step 1: 确认需求
向用户确认：
- 目标接口（URL 或 API 名称）
- 测试场景（发送消息、获取数据、上传文件等）
- 是否需要截图官方前端页面

### Step 2: 接口测试
根据接口类型选择测试方式：

**公开接口（无需认证）：**
- 直接使用 web_fetch 或 exec(curl) 测试
- 记录响应状态码、返回数据结构

**需认证接口：**
- 指导用户获取测试 Token（沙箱环境）
- 使用测试凭证发送请求
- 记录认证流程复杂度

**仅文档接口：**
- 分析文档完整性
- 评估实现难度
- 标注潜在风险点

### Step 3: 前端页面截图（如需要）
使用 browser-use 技能：
- 打开官方文档/示例页面
- 截取接口配置 UI、参数表单、响应示例
- 标注关键区域

### Step 4: 输出可行性报告

## 注意事项

1. **沙箱优先** — 优先使用沙箱/测试环境，避免影响生产数据
2. **凭证安全** — 测试 Token 不记录到日志，用后销毁
3. **限流感知** — 测试频率控制在 API 限流范围内
4. **错误处理** — 记录所有错误码和错误信息
5. **中文翻译** — 英文文档/返回需翻译关键字段

## 依赖技能

- browser-use（浏览器自动化截图）
- web_fetch（网页抓取）
- searxng（搜索测试示例）

## 示例场景

### 用户问："帮我测试 Meta Messenger 发送消息接口能不能用"

**执行：**
1. 搜索 Meta Messenger API 沙箱环境
2. 指导用户创建测试应用获取 Token
3. 发送测试请求（文本消息）
4. 记录返回参数和状态
5. 截取官方配置页面
6. 输出可行性报告

### 用户问："这个接口的前端页面是什么样的"

**执行：**
1. 打开官方文档/控制台
2. 使用 browser-use 截图
3. 标注关键 UI 区域
4. 返回截图 + 说明

## 技能边界

**能做：**
- ✅ 测试公开接口可用性
- ✅ 指导获取测试凭证
- ✅ 截取官方前端页面
- ✅ 输出可行性评估报告
- ✅ 提供替代方案建议

**不能做：**
- ❌ 绕过认证或限流
- ❌ 访问需要用户授权的私有数据
- ❌ 代替开发团队实现接口
- ❌ 保证接口 100% 稳定（以官方 SLA 为准）
